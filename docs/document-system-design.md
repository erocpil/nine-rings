# Nine Rings 文档管理系统设计文档

## 1. 问题与动机

### 1.1 背景

Nine Rings 最初是一个按日期组织的轻量笔记 + 待办应用，数据模型以 `date`（日期）为第一维度。所有笔记属于特定日期，通过标签辅助分类。

这种设计适合"碎片记录"场景，但存在三个问题：

1. **笔记和文档的概念混淆**：随笔（当日碎碎念）和结构化文档（项目设计说明、API 参考手册、操作指南）共享同一存储和 UI，但两者的生命周期、组织方式、使用场景截然不同
2. **"该放哪里"的决策成本**：写一篇新文档时，用户面临"应该放在哪个日期下？加什么标签？"的决策负担，容易导致随意放置或干脆不写
3. **检索维度单一**：只能按日期翻阅或全文搜索，无法按主题层级浏览，也无法按文档类型过滤

### 1.2 目标

在不破坏现有随笔功能的前提下，新增一个**文档管理子系统**，满足以下需求：

| 需求 | 说明 |
|------|------|
| **结构化存放** | 以层级目录组织，文档归属清晰 |
| **多维度分类** | 不只靠目录，还能按文档类型和概念标签查找 |
| **低决策成本** | 新建文档只需选一个目录，不纠结标签和类型 |
| **批量管理** | 导入、删除、只读切换支持批量操作 |
| **向后兼容** | 现有日记/随笔不受影响，零数据迁移 |

## 2. 分类模型：三维正交分类

文档管理系统采用**三个独立且正交的分类维度**：

```
         目录（生命周期）
         ┌──────────────────┐
         │  P.A.R.A. 五目录  │  ← 并存放在哪里（决策一次）
         │  Projects/Areas/  │
         │  References/Ideas/│
         │  Archives         │
         └────────┬─────────┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
    ▼             ▼             ▼
 类型            概念           关联
(Diátaxis)   (Zettelkasten)  (双向链接)
```

### 2.1 维度一：目录（生命周期，P.A.R.A.）

**决策原则**：一篇文档只归属一个目录。这个目录表达文档的**生命周期阶段和行动属性**，不是文件夹式的静态分类。

| 目录 | 含义 | 存放内容 |
|------|------|---------|
| **Projects** | 活跃项目 | 有明确截止日期/交付物的项目文档 |
| **Areas** | 持续领域 | 需要长期维护、没有终点的领域（如健康、运维） |
| **References** | 参考资料 | 外部信息整理（技术文档、读书笔记） |
| **Ideas** | 缓冲想法 | 未成型的想法，待孵化/决策 |
| **Archives** | 归档 | 已完成或废弃的项目/资源 |

**目录间移动**：文档的 `storagePath` 可以通过属性面板修改。随着文档生命周期变化，从 Ideas → Projects → Archives 自然流转。

**子目录**：每级目录下允许再分子目录，如 `projects/nine-rings/docs` 表示 Nine Rings 项目下的 docs 子模块。

**存放决策 VS 查找维度**：这是设计的核心 trade-off —— **目录只管"放"，不管"找"**。用户创建文档时只做一件事（选目录），查找时可以同时用目录、类型、概念三个维度交叉过滤，不依赖当初的存放位置。

### 2.2 维度二：类型（Diátaxis）

Diátaxis 是一种技术文档分类框架，关注**文档服务于用户的哪种认知需求**：

| 类型 | 用户意图 | 典型示例 |
|------|---------|------|
| **Explanation** (解释) | 理解原理 | 架构设计说明、"为什么这样设计" |
| **How-to Guide** (指南) | 完成具体操作 | 部署步骤、配置指南 |
| **Reference** (参考) | 查阅具体信息 | API 参数表、配置项手册 |
| **Tutorial** (教程) | 从头学完 | 新手入门引导 |

同一主题可以有四种类型的文档，如 DPDK 主题下可以同时存在"DPDK 内存管理原理（解释）"和"DPDK 编译步骤（指南）"。

### 2.3 维度三：概念（Zettelkasten）

**概念**是跨目录、跨类型的关联维度。一篇文档可以关联多个概念标签（如 `DPDK`、`P4`、`eBPF`）。

概念用于：
- **聚合视图**：在某目录下查看"所有与 `DPDK` 相关的文档"
- **跨主题连接**：一篇 P4 文档关联 `parse graph` 概念，一篇 eBPF 文档也关联同样概念 → 产生意外连接
- **三维搜索**：目录 + 类型 + 概念任意组合过滤

### 2.4 三维正交的设计约束

| 约束 | 理由 |
|------|------|
| 目录只有一个 | 降低存放决策成本，避免"该放 A 还是 B"的纠结 |
| 类型只有一个 | Diátaxis 四种类型互斥，一篇文档不可能同时是教程又是参考 |
| 概念可以有多个 | 概念是标签性质，允许自由组合多标签 |
| 目录决定生命周期，其他维度用于查找 | 一次决策存放，多维检索 |

## 3. 数据模型

### 3.1 Note 模型扩展

文档复用现有 `Note` 类型，在原有字段基础上新增四个字段：

```
Note {
  // ── 原有字段（随笔）──
  id: string
  date: string           // 所属日期
  title: string
  content: DeltaOps      // Quill Delta 格式
  tags: string[]         // 随笔标签（与文档概念独立）
  pinned: boolean
  sort_order: number
  readonly: boolean
  created_at: string     // ISO 8601
  updated_at: string
  deleted_at: string?    // 软删除标记

  // ── 新增字段（文档分类）──
  storagePath?: string   // 文档路径，如 "projects/nine-rings"
  docType?: DocType      // Diátaxis 类型
  concepts?: string[]    // Zettelkasten 概念标签
  linkedDocIds?: string[] // 双向关联文档 ID 列表
}
```

**关键设计决策**：不创建独立的 `Document` 表。原因：

1. **统一编辑器**：文档和随笔共用 TipTap 编辑器，内容格式相同（Quill Delta）
2. **统一搜索**：全文搜索无需跨表
3. **渐进迁移**：随笔可以逐步转为文档（追加 `storagePath` 即为文档）
4. **简单性**：一张表比两张表 + 联合查询简单得多

**判定规则**：`storagePath` 是否存在，区分文档（有）和随笔（无）。

### 3.2 索引设计（IndexedDB）

```
notes 表索引：
  date        → 按日期查询随笔
  storagePath → 按路径查询文档
  tags        → 按标签查询
  search_text → 全文搜索
```

IndexedDB 不支持真正的全文索引，`search_text` 列存储内容 + 标题 + 标签的纯文本拼接，搜索时用字符串 `includes` 匹配。

### 3.3 存储路径格式

```
storagePath = "<root>/[subdir/...]"

示例:
  "projects"                          → 根目录
  "projects/nine-rings"               → 二级目录
  "references/dpdk/memory-management" → 三级目录
```

目录分隔符固定为 `/`。子路径创建时自动规范化（非字母/数字/汉字的字符替换为 `-`，连续 `-` 合并为单个）。

### 3.4 概念序列化

IndexedDB 不能直接存储数组，`concepts` 字段存储为 JSON 字符串（`JSON.stringify`），读取时 `JSON.parse` 还原。IDB adapter 封装了此转换逻辑，上层只操作数组。

### 4.2 API 设计

所有对文档的操作用 `api.docs.*` 命名空间：

```typescript
api.docs = {
  tree(): PathNode[]              // 构建文档树
  listByPath(prefix): Note[]      // 按路径列出文档
  search(query): Note[]           // 三维搜索
  allConcepts(): string[]         // 获取所有已有概念
}
```

文档的 CRUD 操作复用 `api.notes.*` 接口（create/update/delete），因为文档和笔记共用 `Note` 表。

### 4.3 路径树构建算法（`getPathTree`）

这是文档系统最核心的算法。输入是所有带 `storagePath` 的 Note，输出是 `PathNode[]` 数组。

```
算法步骤:

1. 遍历所有文档（未删除 + 有 storagePath）
   a. 为文档生成 PathNode { path: "<storagePath>/<id>", type: "document", name: <title> }
   b. 为文档路径的每一级前缀生成文件夹节点
      例如 "projects/nine-rings/docs" 生成三个文件夹节点:
        "projects"
        "projects/nine-rings"
        "projects/nine-rings/docs"

2. 统计每个文件夹下文档数（count 字段）
   注意: 计数包含子目录，如 "projects" 的 count = 其下所有子目录文档总数

3. 前端渲染时按 `/` 分割 path，构建父子关系树
   第一级元素为根节点 -> 按 childrenMap 递归渲染
```

**文档节点唯一性**：path 使用 `storagePath/id` 而非 `storagePath/title`，因为 title 可能重复。

**文件夹节点中间层级**：循环条件 `i <= parts.length`（而非 `i < parts.length`），确保为完整路径（包括末级文件夹）创建文件夹节点。这是在实际调试中发现的 bug（`getPathTree` 缺少中间文件夹节点导致子文档成为孤儿无法在树上显示）。

## 5. 组件设计

### 5.1 DocTree（文档树）

位置：侧边栏"📂 文档"Tab 下。

功能：
- **树形结构**：按路径层级渲染文件夹 + 文档，默认展开第一级
- **折叠**：文件夹节点前有 ▶/▼ 箭头，点击折叠/展开
- **点击文件夹**：打开 MOC 视图（该目录下所有文档的表格）
- **点击文档**：在主编辑区打开该文档
- **选中高亮**：三层叠加 —— 3px 左侧竖线 + 深色背景 + 粗体主题色标题
- **右键菜单**：文档节点上右键弹出菜单（重命名、切换只读、删除）
- **重命名**：内联输入框，支持 Enter 确认、Esc 取消、失焦提交
- **批量模式**：header 中 ☐ 按钮进入批量选择，文档前出现 checkbox，支持批量删除、批量只读
- **属性面板开关**：header 中 ⊟/⊞ 按钮，控制打开文档时是否自动弹出属性面板
- **缩进**：8px/层，保持紧凑

关键交互：

```
单击文件夹 → 打开 MOC 视图（占据主编辑区）
单击文档   → 打开文档编辑器
右键文档   → 弹出上下文菜单
☐ 按钮    → 进入批量选择模式
⊟ 按钮    → 关闭属性面板（变为 ⊞，文档不再自动弹出属性）
```

### 5.2 DocMOC（Map of Content）

MOC（Map of Content）是某目录下所有文档的表格视图。

功能：
- **面包屑导航**：显示当前目录路径
- **表格列**：标题、类型（彩色 badge）、概念（chip，最多显示 3 个 + 省略计数）、关联数、更新时间
- **排序**：按 `updated_at` 倒序
- **点击行**：打开对应文档
- **空态**：`此目录下暂无文档`

### 5.3 DocCreateDialog（新建文档）

功能：
- **标题输入**：自动聚焦
- **位置选择**：下拉框选择 P.A.R.A. 根目录 + 可选手动输入子路径
- **路径预览**：实时显示完整 `storagePath`
- **类型选择**：四个按钮（解释/指南/参考/教程），单选
- **概念标签**：输入 + Enter 添加，自动补全已有概念，可删除
- **提交**：Enter 快捷键或点击按钮

### 5.4 PropertiesPanel（属性面板）

功能：
- **位置显示**：只读显示当前文档路径，根目录带 emoji 图标，子路径去除根路径前缀避免重复
- **类型切换**：四个按钮，点击即时修改
- **概念管理**：输入添加/删除概念，自动补全
- **文档关联**：搜索并关联其他文档（双向链接），显示已关联文档列表

面板关闭逻辑：
- 点 ✕ 按钮关闭
- 切换到随笔（非文档）时关闭
- `propertiesAutoShow` 关闭时，打开文档不弹出面板

## 6. 批量导入

### 6.1 `scripts/md-to-nine-rings.py`

将目录中的 `.md` 文件批量导入为文档。

```bash
# 基本用法：导入到 references/dpdk
python3 scripts/md-to-nine-rings.py --serve --path references/dpdk ./dpdk-docs/

# 指定类型和概念
python3 scripts/md-to-nine-rings.py --serve --path projects/nine-rings \
    --type reference --concepts DPDK,P4,tunnel ./docs/
```

参数说明：

| 参数 | 作用 | 默认值 |
|------|------|--------|
| `--serve` | 通过 HTTP POST 推送到 dev server，浏览器直接看到结果 | 不指定则生成 JSON 文件 |
| `--path` | 文档目标存放路径 | 从目录名推断：`references/<目录名>` |
| `--type` | Diátaxis 类型 | 不指定 |
| `--concepts` | 逗号分隔的概念标签 | 不指定 |

### 6.2 Markdown 解析

导入脚本内置了 Markdown → Quill Delta 转换器，支持：

| Markdown | Delta 输出 |
|----------|-----------|
| `# ## ###` 标题 | `{ header: 1/2/3 }` |
| `**粗体**` | `{ bold: true }` |
| `*斜体*` | `{ italic: true }` |
| `` `代码` `` | `{ code: true }` |
| ` ``` 代码块 ``` ` | `{ 'code-block': true }` |
| `- 无序列表` | `{ list: 'bullet' }` |
| `1. 有序列表` | `{ list: 'ordered' }` |
| `> 引用` | `{ blockquote: true }` |
| `[链接](url)` | `{ link: 'url' }` |
| `--- 分割线` | 横线字符 × 8 + 删除线样式 |

## 7. 关键设计决策与权衡

### 7.1 为什么不创建独立的 Document 表？

**决定**：复用 Note 表，用 `storagePath` 字段区分。

**理由**：
- 避免两套 CRUD 代码
- 编辑器、搜索、导出/导入逻辑零改动
- 按日期的随笔可以自然"晋升"为文档
- IndexedDB 没有 JOIN，两表联合查询性能差

**代价**：
- Note 对象可能包含不相关字段（文档有 `concepts`，随笔有 `date`）
- 需要在多处判断 `storagePath` 来控制分支逻辑

### 7.2 为什么不把目录做成关系型树表？

**决定**：目录是虚拟构造的（从 `storagePath` 字符串中解析），不是持久化的独立实体。

**理由**：
- 目录不需要独立存储任何状态（无元数据、无权限、无排序）
- 避免同步复杂度（树节点与文档节点的一致性）
- `getPathTree()` 每次实时构建，保证与文档数据强一致

**代价**：
- 空目录无法显示（没有文档时树为空）
- 文件夹重命名需要更新所有子文档的 `storagePath`（批量操作，当前未实现）

### 7.3 类型和概念为什么在创建时可选？

**决定**：创建文档时只强制填标题和目录，类型和概念为可选。

**理由**：降低创建门槛。用户可以先写内容，随后在属性面板补充分类维度。这遵循"先写后分类"的理念 —— 不让分类决策阻碍创作流程。

### 7.4 概念自动补全为什么用全量拉取？

**决定**：`api.docs.allConcepts()` 返回所有已有概念（Set 去重），客户端做前缀匹配补全。

**理由**：
- 概念数量通常不会超过几百个，全量拉取性能可接受
- 避免维护概念索引表（增加复杂度）
- 客户端补全即时响应，无网络延迟

**代价**：
- 概念数量极大（数千）时性能下降 —— 当前规模不构成问题

### 7.5 属性面板显隐为什么存 localStorage 而非后端配置？

**决定**：`propertiesAutoShow` 状态存储在 `localStorage`，不存入 `Config` 表。

**理由**：这是一个 UI 状态偏好，不是应用级配置。不需要跨设备同步（换设备后默认打开即可）。

### 7.6 文件夹点击为什么是目录名而非 ▶ 箭头？

**决定**：点击文件夹名称 → 打开 MOC 视图；点击 ▶/▼ → 折叠/展开。

**理由**：
- 分离"导航"和"结构操作"两个动作
- 符合文件管理器直觉（点击文件夹名进入，点击三角展开）
- 如果合二为一会导致交互歧义

## 10. 后续演进方向

### 10.1 概念聚合页

点击某概念标签 → 打开概念聚合页，列出所有关联该概念的文档。类似 MOC 但跨目录、跨类型。

### 10.2 双向链接自动建议

在编辑器中输入 `[[` 时自动弹出文档搜索补全，选择后插入双向链接卡片。被引用的文档自动追加 `linkedDocIds`。

### 10.3 文件夹操作

- 文件夹重命名：批量更新所有子文档的 `storagePath` 前缀
- 文件夹移动：同上，core logic 是路径前缀替换
- 空文件夹显示：允许创建空文件夹（当前无文档时树为空）

### 10.4 随笔与文档统一

在 `storagePath` 推出后，每日笔记本质上就是 `daily/<YYYY-MM-DD>` 下的文档。可以逐步将 `DailyPage` 废弃，统一用 `Note` + `storagePath` 表达。随笔的"按日期排列"视图变成 `daily/` 路径下按时间排序的特殊视图。
